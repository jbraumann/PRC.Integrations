## 1. Overview & Architecture

Parametric Robot Control (PRC) is a client–server system for simulating and controlling industrial robots. The **PRC Server** runs locally (default `https://127.0.0.1:5001`) and exposes a **gRPC** API over HTTPS with TLS. Any language with gRPC support can act as a client. For .NET users, PRC additionally provides a high-level **Client wrapper library** in the `PRC.GRPC.Client` namespace that handles connection management, serialization, and event-based feedback — see [Section 4](#4-net-client-library-prcgrpcclient).

```
┌─────────────────────┐         gRPC / HTTPS         ┌──────────────────┐
│   Client Plugin     │  ◄──────────────────────────► │    PRC Server    │
│  (GH, Blender,      │    SetupRobot                 │                  │
│   Unity, Fusion360,  │    AddRobotTask               │  Robot Library   │
│   Python, JS, …)    │    SubscribeRobotFeedback      │  Kinematics      │
│                     │    GetSimulatedRobotState      │  Code Generation │
│                     │    UpdateVariable              │  Collision Check │
│                     │    SendPing                    │  Real-time Ctrl  │
└─────────────────────┘                               └──────────────────┘
```

**Key principles:**
- **Units:** PRC works in **millimetres** for all positional data and **degrees** for all angular/axis data.
- **Coordinate system:** Right-handed, **Z+ = up**. All transformation matrices are expressed relative to WorldXY in this convention.
- **Communication:** GRPC over HTTPS with TLS. Supports gzip compression and unlimited message sizes.
- **Protobuf definition:** The PRC API is defined in a single `.proto` file that contains all messages, enums, and the service contract. Code can be generated for any language using the standard `protoc` compiler.

## 2. Lifecycle & Data Flow

Every PRC integration follows the same three-phase lifecycle. Understanding this lifecycle is essential before diving into individual messages.

### The Three Phases

```
Phase 1: SETUP          Phase 2: TASK              Phase 3: SIMULATION
─────────────────       ──────────────────          ────────────────────
  SetupRobot()     →      AddRobotTask()       →     GetSimulatedRobotState()
                                                      SubscribeRobotFeedback()
  Defines:               Defines:                    Queries:
  - Robot model          - Motion commands            - Robot state at 0.0–1.0
  - Robot driver         - Actions (I/O, hold…)       - Joint transformations
  - Tools                - Flow control               - TCP position
  - Base                 - Task type                  - Axis alarms
  - Collision geo        Returns:
  - External axes        - SimulationResult
                         - Generated robot code
  Returns:               - Reachability feedback
  - Server robot ID
  - Settings dictionary
```

### Pseudocode — Full Lifecycle

```
// ── Phase 1: Setup ──
robot_id = SetupRobot(
    client_id    = "my_unique_id",
    robot_model  = "KUKA.KUKA_KR610R11002",   // preset from the robot library
    robot_driver = "KUKA.KSS_KRL_Driver",
    tools        = { "0": Tool(tcp = identity_frame) },
    base         = Base(frame = world_origin),
    collision    = [],
    external_axes = []
)
// → Returns: server-assigned robot_id, settings dictionary

// ── Phase 2: Task ──
simulation_result = AddRobotTask(
    id       = robot_id,
    task     = Task(
        type    = SIMULATE_AND_EXECUTE,
        payload = [
            MotionGroup(
                type     = PTP,
                tool_id  = "0",
                commands = [
                    AxisMotion(target = [-45, -90, 90, 0, 0, 0], speed = 0.15),
                    AxisMotion(target = [ 45, -90, 90, 0, 0, 0], speed = 0.15),
                ]
            )
        ]
    ),
    settings = settings   // from Phase 1, optionally modified
)
// → Returns: SimulationResult with axis data, reachability, code

// ── Phase 3: Visualization ──
for slider_value in 0.0 to 1.0:
    robot_state = GetSimulatedRobotState(
        id               = robot_id,
        normalized_state = slider_value
    )
    // robot_state.robot_transformations → one Matrix4x4 per joint
    // Apply each matrix to the corresponding joint mesh
```

### When to Call SetupRobot Again

`SetupRobot` establishes the simulation environment. You **must** call it again when any of the following change:

| Change | Requires new SetupRobot? |
|---|---|
| Different robot model (e.g., switching from KR10 to KR120) | **Yes** |
| Different robot driver (e.g., switching from KSS_KRL to UR_Driver) | **Yes** |
| Modifying the tool dictionary (adding/removing/changing tools) | **Yes** |
| Changing the initial base frame | **Yes** |
| Adding or removing external axes | **Yes** |
| Modifying collision geometry | **Yes** |
| Changing motion commands (different waypoints, speeds) | No — use `AddRobotTask` |
| Adjusting settings values (e.g., interpolation parameters) | No — pass new settings in `AddRobotTask` |
| Moving the simulation slider (querying different toolpath positions) | No — use `GetSimulatedRobotState` |

### ID Reuse Behaviour

- The `client_id` in `SetupRobotRequest` is chosen by your integration. The server returns a `robot_id` in the reply.
- **If you call `SetupRobot` again with the same `client_id`, the previous robot setup is discarded and replaced.** The server does not maintain multiple robots under the same ID.
- To run multiple robots simultaneously, use different `client_id` values for each.
- The returned `robot_id` (from `SetupRobotReply.id`) is used in all subsequent `AddRobotTask`, `GetSimulatedRobotState`, `SubscribeRobotFeedback`, and `UpdateVariable` calls.

### When to Call AddRobotTask Again

`AddRobotTask` can be called **repeatedly** on the same robot setup whenever:
- Motion commands change (new waypoints, speeds, motion types)
- Actions change (I/O, hold times, code insertion)
- Settings are modified (pass updated settings dictionary)
- You want to re-simulate with different parameters

Each call returns a fresh `SimulationResult`. The settings dictionary can be modified between calls without requiring a new `SetupRobot`.

### SubscribeRobotFeedback — Continuous Stream

After setup, `SubscribeRobotFeedback` opens a **server-streaming** connection that continuously delivers:
- **Heartbeats** — periodic alive signals
- **RobotState updates** — when `GetSimulatedRobotState` is called with `stream_update=true`, or during real-time robot control
- **Settings updates** — when the server modifies settings (e.g., during real-time control)
- **Pings** — response to ping requests

This stream stays open for the lifetime of the robot setup. It is automatically started by the .NET Client library when `SetupRobot` is called.

## 3. Connection Setup

The PRC Server listens on `https://127.0.0.1:5001` by default. Connections use TLS — a `PRCServerCertificate.pem` file is distributed with the server for SSL verification. If the certificate is unavailable, clients can disable certificate validation (not recommended for production).

### Channel Configuration Recommendations

| Setting | Recommended Value |
|---|---|
| Max receive message size | Unlimited (`null` / `-1`) |
| Max send message size | Unlimited (`null` / `-1`) |
| Compression | Gzip (Fastest) |
| Retry policy | 5 attempts, 1s initial backoff, 3s max, 1.5x multiplier |
| Retryable status codes | `Unavailable` |

### C# Raw GRPC Connection

```csharp
string ip = "https://127.0.0.1:5001";

var defaultMethodConfig = new MethodConfig
{
    Names = { MethodName.Default },
    RetryPolicy = new RetryPolicy
    {
        MaxAttempts = 5,
        InitialBackoff = TimeSpan.FromSeconds(1),
        MaxBackoff = TimeSpan.FromSeconds(3),
        BackoffMultiplier = 1.5,
        RetryableStatusCodes = { StatusCode.Unavailable }
    }
};

var grpcChannel = GrpcChannel.ForAddress(ip, new GrpcChannelOptions
{
    ServiceConfig = new ServiceConfig { MethodConfigs = { defaultMethodConfig } },
    MaxReceiveMessageSize = null,
    MaxSendMessageSize = null,
    CompressionProviders = new List<ICompressionProvider>()
    {
        new Grpc.Net.Compression.GzipCompressionProvider(
            System.IO.Compression.CompressionLevel.Fastest)
    }
});

var client = new ParametricRobotControlService.ParametricRobotControlServiceClient(grpcChannel);
```

## 4. .NET Client Library (PRC.GRPC.Client)

For **.NET users**, PRC provides a high-level wrapper in the `PRC.GRPC.Client` namespace that significantly simplifies integration. Instead of manually constructing GRPC messages, serializing data, and managing streaming calls, you work with native .NET objects from the `PRC.Core` namespace and the `Client` class handles all communication.

### Why Use the .NET Client Library?

| Feature | Raw GRPC | .NET Client Library |
|---|---|---|
| Connection management | Manual channel setup, certificate handling | `client.Connect(ip)` — handles TLS, retry, compression automatically |
| Robot setup | Manual message construction | Pass `IRobotProperties` objects directly; library serializes to GRPC |
| Feedback streaming | Manual async stream reading | Event-based: `RobotStateUpdatedEventHandler`, `RobotSettingsUpdatedEventHandler` |
| Simulation state | Manual `GetSimulatedRobotStateAsync` | `client.UpdateRobot(0.5f)` — returns deserialized `RobotState` |
| Task submission | Manual message construction | `client.AddTask(task, settings)` — returns `SimulationFeedback` |
| Robot models | String class names only | Direct instantiation: `new PRC.Library.Robots.KUKA.KUKA_KR610R11002()` |
| Reconnection | Manual | `client.Reconnect()` — re-establishes connection, setup, and task |

### Prerequisites

Reference the PRC NuGet packages or DLLs:
- `PRC.GRPC` — contains the `Client` class and GRPC plumbing
- `PRC.Core` — data types (`Tool`, `Base`, `Task`, `JointTarget`, `RobotState`, etc.)
- `PRC.Library` — built-in robot models and drivers

### Client API Reference

The `PRC.GRPC.Client.Client` class exposes the following methods and properties:

| Method | Signature | Description |
|---|---|---|
| `Connect` | `async Task<TaskFeedback> Connect(string ip, bool grpcWeb = false)` | Establish connection to PRC server. Handles TLS certificates automatically. |
| `SetupRobot` | `async Task<SetupFeedback> SetupRobot(string clientId, IRobotProperties robot, string robotDriver)` | Initialize robot environment. Automatically subscribes to the feedback stream. Returns settings dictionary. |
| `AddTask` | `async Task<SimulationFeedback> AddTask(Task task, Dictionary<string, string> settings)` | Send motion commands. Returns simulation result with code, axis data, reachability. |
| `UpdateRobot` | `async Task<RobotState> UpdateRobot(float simulationState, bool streamFeedback = false)` | Query robot state at a normalized position (0.0–1.0). |
| `UpdateVariable` | `async Task<Dictionary<string, List<Variable>>> UpdateVariable(Variable variable)` | Set/update a robot variable. Returns all variables of all connected robots. |
| `Ping` | `async Task<Ping> Ping(string payload = "")` | Connection health check. |
| `Disconnect` | `async Task<TaskFeedback> Disconnect()` | Close the connection gracefully. |
| `Reconnect` | `async Task<TaskFeedback> Reconnect()` | Reconnect, re-setup robot, and re-send last task. |

| Property | Type | Description |
|---|---|---|
| `robotState` | `PRC.Core.Primitives.RobotState` | The most recent robot state received from the server. |
| `clientID` | `string` | The server-assigned robot ID. |
| `actSettings` | `Dictionary<string, string>` | Current settings dictionary. |
| `connectivityState` | `ConnectivityState` | Current GRPC channel state. |

### Event Handlers

The Client library automatically subscribes to the `SubscribeRobotFeedback` stream when `SetupRobot` is called. Events are raised on the calling thread:

| Event | EventArgs Type | Fired When |
|---|---|---|
| `RobotStateUpdatedEventHandler` | `RobotStateUpdatedEventArgs` (`.RobotState`) | A new robot state is received (simulation update, real-time control). |
| `RobotSettingsUpdatedEventHandler` | `RobotSettingsUpdatedEventArgs` (`.RobotSettings`) | Settings are updated by the server. |
| `PingEventHandler` | `RobotPingEventArgs` (`.PingPayload`) | A ping response is received. |

Call `client.ClearEvents()` to unsubscribe all handlers.

### Pseudocode — .NET Client Lifecycle

```
client = new Client()

// 1. Connect
feedback = client.Connect("https://127.0.0.1:5001")
if feedback.Status != Success: abort

// 2. Subscribe to events
client.RobotStateUpdatedEventHandler += OnRobotStateUpdated
client.RobotSettingsUpdatedEventHandler += OnRobotSettingsUpdated

// 3. Create robot from library
robot = new PRC.Library.Robots.KUKA.KUKA_KR610R11002()
robot.ToolDictionary = { "0": new Tool() }
robot.InitialBase = new Base()

// 4. Setup
setupFeedback = client.SetupRobot("my_id", robot, "KUKA.KSS_KRL_Driver")
settings = setupFeedback.Settings   // Dictionary<string, string>

// 5. Build task
task = new Task()
task.TaskType = SimulateAndExecuteTask
motionGroup = new PTPMotionGroup()
motionGroup.ToolID = "0"
motionGroup.PTPMotions = [
    AxisMotion(target = [-45, -90, 90, 0, 0, 0], speed = 0.15),
    AxisMotion(target = [ 45, -90, 90, 0, 0, 0], speed = 0.15),
]
task.Commands.Add(motionGroup)

// 6. Send task
simFeedback = client.AddTask(task, settings)
// simFeedback.Result.Code → generated KRL
// simFeedback.Result.IsValid → true if no issues
// simFeedback.Result.Time → estimated seconds

// 7. Animate simulation
for i = 0 to 100 step 3:
    client.UpdateRobot(i / 100.0, streamFeedback = true)
    // → triggers RobotStateUpdatedEventHandler with new transforms
    sleep(500ms)

// 8. Cleanup
client.ClearEvents()
client.Disconnect()
```

### Full .NET Client Library Example

```csharp
using PRC.GRPC.Client;
using PRC.Core.Classes;
using Task = System.Threading.Tasks.Task;

// 1. Connect
Client client = new Client();
var connectFeedback = await client.Connect("https://127.0.0.1:5001");

if (connectFeedback.Status == Status.Success)
{
    // 2. Subscribe to events
    client.RobotStateUpdatedEventHandler += (sender, e) =>
    {
        if (e.RobotState != null)
        {
            var axes = e.RobotState.AxisPosition.AxisValues;
            Console.WriteLine($"State: A1={axes[0]:F1} A2={axes[1]:F1} A3={axes[2]:F1}");
        }
    };
    client.RobotSettingsUpdatedEventHandler += (sender, e) =>
    {
        Console.WriteLine("Settings updated: " + e.RobotSettings.Count + " entries");
    };

    // 3. Create robot from library (includes geometry, kinematics, axis limits)
    var robot = new PRC.Library.Robots.KUKA.KUKA_KR610R11002();
    robot.ToolDictionary = new Dictionary<string, PRC.Core.Classes.Tool>
    {
        ["0"] = new PRC.Core.Classes.Tool()  // Identity TCP at flange
    };
    robot.InitialBase = new PRC.Core.Classes.Base();

    // 4. Setup robot
    var setupFeedback = await client.SetupRobot("my_unique_id", robot, "KUKA.KSS_KRL_Driver");
    var settings = setupFeedback.Settings;  // Dictionary<string, string>
    Console.WriteLine("Settings: " + string.Join(", ",
        settings.Select(kv => $"{kv.Key}={kv.Value}")));

    // 5. Build task with PTP motion group
    var robotTask = new PRC.Core.Commands.Task();
    robotTask.TaskType = PRC.Core.Primitives.Enums.TaskType.SimulateAndExecuteTask;
    robotTask.Name = "MyTask";

    var ptpGroup = new PRC.Core.Commands.Motion.Groups.PTPMotionGroup();
    ptpGroup.Base = new PRC.Core.Classes.Base();
    ptpGroup.ToolID = "0";
    ptpGroup.Interpolation = "C_PTP";
    ptpGroup.PTPMotions = new PRC.Core.Interfaces.IMotion[]
    {
        new PRC.Core.Commands.Motion.Axis
        {
            Target = new PRC.Core.Primitives.JointTarget
            {
                AxisValues = new float[] { -45, -90, 90, 0, 0, 0 },
                Speed = new float[] { 0.15f }
            }
        },
        new PRC.Core.Commands.Motion.Axis
        {
            Target = new PRC.Core.Primitives.JointTarget
            {
                AxisValues = new float[] { 45, -90, 90, 0, 0, 0 },
                Speed = new float[] { 0.15f }
            }
        }
    };
    robotTask.Commands.Add(ptpGroup);

    // 6. Send task and get simulation result
    var simFeedback = await client.AddTask(robotTask, settings);
    Console.WriteLine($"Valid: {simFeedback.Result.IsValid}");
    Console.WriteLine($"Time: {simFeedback.Result.Time:F2}s");
    Console.WriteLine($"KRL Code:\n{simFeedback.Result.Code}");

    // 7. Animate simulation (0% → 100%)
    for (int i = 0; i <= 100; i += 3)
    {
        await client.UpdateRobot(i / 100f, streamFeedback: true);
        await Task.Delay(500);
    }

    // 8. Cleanup
    client.ClearEvents();
    await client.Disconnect();
}
```

## 5. GRPC Service Definition

The PRC protobuf definition exposes **6 RPC methods** in the `ParametricRobotControlService`:

```protobuf
service ParametricRobotControlService {
  // Step 1: Setup the simulation/control environment.
  rpc SetupRobot (SetupRobotRequest) returns (SetupRobotReply);

  // Step 2: Define a task to be processed or simulated.
  rpc AddRobotTask (AddRobotTaskRequest) returns (AddRobotTaskReply);

  // Step 3: Subscribe to feedback from the simulation/control environment.
  rpc SubscribeRobotFeedback (SubscribeRobotFeedbackRequest) returns (stream RobotFeedback);

  // Step 4: Request a simulation update at a specific progress point.
  rpc GetSimulatedRobotState (GetSimulatedRobotStateRequest) returns (RobotState);

  // Optional: Update a variable. Returns all variables of all connected devices.
  rpc UpdateVariable (UpdateVariableRequest) returns (UpdateVariableReply);

  // Optional: Ping the controller.
  rpc SendPing (Ping) returns (Ping);
}
```

| Method | Type | Purpose |
|---|---|---|
| `SetupRobot` | Unary | Initialize robot model, driver, tools, base, collision geometry. Returns settings dictionary. |
| `AddRobotTask` | Unary | Send motion commands. Returns `SimulationResult` with axis data, reachability, generated code. |
| `SubscribeRobotFeedback` | Server streaming | Real-time stream of heartbeats, robot state, settings updates, pings. |
| `GetSimulatedRobotState` | Unary | Query robot state at a normalized position (0.0–1.0) along the toolpath. |
| `UpdateVariable` | Unary | Set a variable on the robot. Returns all variables of all connected robots. |
| `SendPing` | Unary | Connection health check. |

## 6. Protobuf Message Reference

All messages are defined in the PRC protobuf definition. This section documents every message, enum, and field.

### 6.1 Primitives

#### `Matrix4x4`

A 4x4 homogeneous affine transformation matrix, laid out identically to `System.Numerics.Matrix4x4`. Used for positions, orientations, and joint transformations. **Translation components (m41, m42, m43) are in millimetres.**

Based on the PRC `CoordinateSystem` class, the matrix stores the **coordinate frame axes as rows** and the **origin/translation in the last row**:

```protobuf
message Matrix4x4 {
  // X-Axis (Row 1)
  float m11 = 1;   // X-Axis.X
  float m12 = 2;   // X-Axis.Y
  float m13 = 3;   // X-Axis.Z
  float m14 = 4;   // 0

  // Y-Axis (Row 2)
  float m21 = 5;   // Y-Axis.X
  float m22 = 6;   // Y-Axis.Y
  float m23 = 7;   // Y-Axis.Z
  float m24 = 8;   // 0

  // Z-Axis / Normal (Row 3)
  float m31 = 9;   // Z-Axis.X
  float m32 = 10;  // Z-Axis.Y
  float m33 = 11;  // Z-Axis.Z
  float m34 = 12;  // 0

  // Origin / Translation (Row 4)
  float m41 = 13;  // Origin.X  (mm)
  float m42 = 14;  // Origin.Y  (mm)
  float m43 = 15;  // Origin.Z  (mm)
  float m44 = 16;  // 1 (homogeneous)
}
```

Visually:

```
| m11  m12  m13  m14 |     Row 1: X-Axis direction
| m21  m22  m23  m24 |     Row 2: Y-Axis direction
| m31  m32  m33  m34 |     Row 3: Z-Axis direction (normal)
| m41  m42  m43  m44 |     Row 4: Origin / Translation (mm), 1
                           Col 4: [0, 0, 0, 1] (homogeneous)
```

**Pseudocode:**
```
Matrix4x4:
    m11, m12, m13, m14   // Row 1: X-axis direction vector,  0
    m21, m22, m23, m24   // Row 2: Y-axis direction vector,  0
    m31, m32, m33, m34   // Row 3: Z-axis direction vector,  0
    m41, m42, m43, m44   // Row 4: Origin X(mm), Y(mm), Z(mm), 1
```

**Convention details:** This is a **row-major, row-vector (post-multiply)** layout, matching `System.Numerics.Matrix4x4`. The top-left 3x3 block stores the rotation with each coordinate axis as a row. The translation lives in the **last row** (not the last column), and column 4 is the standard homogeneous `[0, 0, 0, 1]`. This means point transformation uses the **point x matrix** convention (row vector on the left), which is why the translation is in the bottom row rather than the right column.

When porting to engines or libraries that use **column-major layout** or a **pre-multiply / column-vector convention** (e.g., OpenGL, Blender, Fusion 360), you must **transpose** the matrix so the translation ends up in the last column.

#### `CartesianPosition`

Defines a position in 3D Cartesian space. Uses a `oneof` to support three representations:

| Frame Type | Message | Description |
|---|---|---|
| `matrix` | `Matrix4x4` | Full 4x4 transformation matrix. Most general representation. |
| `euler` | `Euler` | Position (X, Y, Z in mm) + orientation (A, B, C) with a specified `EulerFormat`. |
| `cs` | `CoordinateSystem` | Origin point + X-axis vector + Y-axis vector (Z derived from cross product). |

**Pseudocode:**
```
CartesianPosition:
    frame = ONE OF:
        Matrix4x4   → full 4x4 matrix
        Euler       → {x, y, z} in mm + {a, b, c} angles + format
        CoordinateSystem → origin(mm) + x_axis(unit) + y_axis(unit)
    reference = ABSOLUTE | RELATIVE | PARENT
    parent    = optional Matrix4x4 (parent frame, if reference = PARENT)
    id        = string identifier
```

Additional fields:
- **`reference`** (`CartesianReference` enum): `ABSOLUTE` (world frame), `RELATIVE` (offset from current), or `PARENT` (relative to parent matrix).
- **`parent`** (`Matrix4x4`): Optional parent transformation matrix.
- **`id`** (`string`): Identifier for the position.

#### `Euler`

Describes a position and orientation using Euler-like angles:

| Field | Type | Description |
|---|---|---|
| `x` | float | X position in **mm** |
| `y` | float | Y position in **mm** |
| `z` | float | Z position in **mm** |
| `a` | float | First rotation angle in **degrees** |
| `b` | float | Second rotation angle in **degrees** |
| `c` | float | Third rotation angle in **degrees** |
| `format` | `EulerFormat` | Rotation convention (see below) |

**`EulerFormat` enum:**

| Value | Name | Usage |
|---|---|---|
| 0 | `ZYX` | KUKA convention — rotations applied as Z, then Y, then X |
| 1 | `AXISANGLE` | Universal Robots convention — rotation vector (axis * angle) |
| 2 | `RPY` | NEURA convention — Roll-Pitch-Yaw |

**Pseudocode:**
```
Euler:
    x, y, z    = position in mm
    a, b, c    = orientation angles in degrees
    format     = ZYX (KUKA) | AXISANGLE (UR) | RPY (NEURA)
```

#### `CoordinateSystem`

Defines a frame through three vectors:

| Field | Type | Description |
|---|---|---|
| `origin` | `Vector3` | Position of the origin in **mm** |
| `x_axis` | `Vector3` | Direction of the X axis (unit vector) |
| `y_axis` | `Vector3` | Direction of the Y axis (unit vector) |

The Z axis is implicitly derived as the cross product of X and Y (`Z = X x Y`), forming a right-handed coordinate system.

**Pseudocode:**
```
CoordinateSystem:
    origin = Vector3(x_mm, y_mm, z_mm)
    x_axis = Vector3(normalized direction)
    y_axis = Vector3(normalized direction)
    // z_axis = cross(x_axis, y_axis)  ← implicit, right-handed
```

#### `CartesianTarget`

Wraps a `CartesianPosition` with motion parameters:

| Field | Type | Description |
|---|---|---|
| `position` | `CartesianPosition` | The target frame |
| `posture` | `string` | Robot posture/configuration string (e.g., `"110"` for KUKA turn/status) |
| `speed` | `repeated float` | Speed — single value for all axes, or one per axis. Normalized 0.0–1.0. |
| `acceleration` | `repeated float` | Acceleration — single value or one per axis. Normalized 0.0–1.0. |
| `external_axis_values` | `repeated float` | Positions of external axes (in mm or degrees depending on type) |
| `redundancy` | `float` | Redundant axis value (e.g., 7th axis of a 7-DOF robot) |

**Pseudocode:**
```
CartesianTarget:
    position             = CartesianPosition
    posture              = "110"          // robot configuration string
    speed                = [0.15]         // single value or per-axis, 0.0–1.0
    acceleration         = [0.1]          // single value or per-axis, 0.0–1.0
    external_axis_values = [500.0, 0.0]   // mm or degrees
    redundancy           = 0.0            // 7th axis value
```

#### `JointTarget`

Defines a robot position in joint space:

| Field | Type | Description |
|---|---|---|
| `axis_values` | `repeated float` | Joint angles in **degrees** (e.g., 6 values for a 6-axis robot) |
| `speed` | `repeated float` | Speed — single value or one per axis. Normalized 0.0–1.0. |
| `acceleration` | `repeated float` | Acceleration — single value or one per axis. Normalized 0.0–1.0. |
| `external_axis_values` | `repeated float` | External axis positions |

**Pseudocode:**
```
JointTarget:
    axis_values          = [A1_deg, A2_deg, A3_deg, A4_deg, A5_deg, A6_deg]
    speed                = [0.15]         // 15% of max speed
    acceleration         = [0.1]          // 10% of max acceleration
    external_axis_values = [E1, E2, ...]  // mm or degrees
```

#### `Vector3`

A 3D vector with `x`, `y`, `z` float components. Used for positions (in **mm**), directions (unit vectors), and normals.

#### `Int4`

Four integers (`x`, `y`, `z`, `w`). Used for:
- **Mesh faces:** Indices of 4 vertices forming a quad face. For triangles, the last two indices are typically the same (e.g., `{0, 1, 2, 2}`).
- **Mesh colors:** ARGB colour values as integers `{A, R, G, B}` where each is 0–255.

**Pseudocode:**
```
Int4 (as mesh face): { vertex_index_0, vertex_index_1, vertex_index_2, vertex_index_3 }
    // For triangles: vertex_index_2 == vertex_index_3

Int4 (as mesh color): { Alpha, Red, Green, Blue }   // each 0–255
    // Example opaque red: { 255, 255, 0, 0 }
    // Example default gray: { 255, 155, 155, 155 }
```

#### `Variable`

A named variable for robot I/O communication:

| Field | Type | Description |
|---|---|---|
| `name` | `string` | Variable name |
| `boolean` | `bool` | Boolean value (oneof) |
| `single` | `float` | Float value (oneof) |
| `integer` | `int32` | Integer value (oneof) |
| `text` | `string` | String value (oneof) |

Only one of the four value types can be set at a time (protobuf `oneof`).

**Pseudocode:**
```
Variable:
    name  = "GripperOpen"
    value = ONE OF:
        boolean = true
        single  = 3.14
        integer = 42
        text    = "hello"
```

#### `Settings`

A simple dictionary wrapper:

```protobuf
message Settings {
    map<string, string> settings_dictionary = 1;
}
```

The settings are driver-specific key-value pairs (both strings). Returned from `SetupRobotReply` and passed back (optionally modified) in `AddRobotTaskRequest`. See [Section 8](#8-settings-dictionary) for details.

#### `Mesh`

Defines a single geometric mesh:

| Field | Type | Description |
|---|---|---|
| `faces` | `repeated Int4` | Face definitions. Each `Int4` contains indices into the `vertices` array. For triangular faces, `z == w`. |
| `vertices` | `repeated Vector3` | Vertex positions in **mm** |
| `normals` | `repeated Vector3` | Vertex normals (for shading) |
| `mesh_color` | `Int4` | Single colour for the entire mesh as **{A, R, G, B}**, each 0–255 |

**Pseudocode:**
```
Mesh:
    vertices   = [Vector3(x_mm, y_mm, z_mm), ...]
    faces      = [Int4(v0, v1, v2, v3), ...]    // quads; v2==v3 for triangles
    normals    = [Vector3(nx, ny, nz), ...]
    mesh_color = Int4(alpha, red, green, blue)   // 0–255 each
```

#### `PolyMesh`

A compound mesh representing one robotic joint, tool, or environment object. A single `PolyMesh` can contain **multiple visual meshes** (e.g., parts with different colours) and a **separate set of collision meshes**.

| Field | Type | Description |
|---|---|---|
| `meshes` | `repeated Mesh` | **Visualization meshes** — for rendering. Can be any number, with different `mesh_color` per mesh. |
| `collision_convex_hull` | `repeated Mesh` | **Collision meshes** — simplified convex hulls for collision checking. May differ in count and detail from visual meshes. |
| `transform` | `Matrix4x4` | Transformation of this PolyMesh relative to WorldXY (right-handed, Z+ up). |
| `name` | `string` | Identifier for this PolyMesh. |

**Pseudocode:**
```
PolyMesh:
    meshes               = [Mesh, Mesh, ...]   // N visual meshes (different colors)
    collision_convex_hull = [Mesh, Mesh, ...]   // M collision hulls (may differ from N)
    transform            = Matrix4x4            // initial placement
    name                 = "Axis_3"
```

**Key point:** The number of visualization meshes and collision meshes within a single `PolyMesh` are **independent** — a PolyMesh might have 5 visual meshes (for different colored parts) but only 1 simplified collision hull, or vice versa.

#### `TransformationArray`

A list of `Matrix4x4` transformations. Used in `RobotState.robot_transformations` to provide the transformation for each joint of the robot plus the base. See [Section 10](#10-robotstate--visualization) for visualization details.

```protobuf
message TransformationArray {
    repeated Matrix4x4 transformation = 1;
}
```

### 6.2 Robot Data

#### `Robot`

The top-level robot definition, sent in `SetupRobotRequest`:

| Field | Type | Description |
|---|---|---|
| `preset_robot_class` | `string` (oneof `robot_data`) | Reference to a built-in robot. Format: `Manufacturer.ModelClass`, e.g., `"KUKA.KUKA_KR610R11002"`. Maps to `PRC.Library.Robots` namespace. |
| `custom_robot` | `CustomRobot` (oneof `robot_data`) | Full custom robot definition with kinematic chain and geometry. |
| `robot_driver_class` | `string` | Robot driver. Format: `Manufacturer.DriverClass`, e.g., `"KUKA.KSS_KRL_Driver"`. Maps to `PRC.Library.Drivers` namespace. |
| `friendly_id` | `string` | Human-readable name for the robot setup |
| `tool_dictionary` | `map<string, Tool>` | Dictionary of tools keyed by tool ID string. Every `tool_id` referenced in a `MotionGroup` must exist here. |
| `initial_base` | `Base` | The robot's base coordinate system |
| `collision_geometry` | `repeated PolyMesh` | Optional environment collision geometry |
| `external_axes` | `repeated ExternalAxis` | External axes attached to the robot |
| `cell_geometry` | `PolyMesh` | Optional cell geometry for visualization and collision |
| `data` | `MetaData` | Additional data |

**Pseudocode:**
```
Robot:
    robot_data = ONE OF:
        preset_robot_class = "KUKA.KUKA_KR610R11002"   // use built-in model
        custom_robot       = CustomRobot(...)            // or define custom kinematic chain
    robot_driver_class = "KUKA.KSS_KRL_Driver"
    friendly_id        = "My Robot"
    tool_dictionary    = { "0": Tool(...), "1": Tool(...) }
    initial_base       = Base(frame = world_origin)
    collision_geometry = [PolyMesh, PolyMesh, ...]       // environment obstacles
    external_axes      = [ExternalAxis, ...]
    cell_geometry      = PolyMesh(...)                   // work cell structure
```

**The `robot_data` oneof:** You must set **either** `preset_robot_class` (to use a built-in model) **or** `custom_robot` (to define a custom kinematic chain). Setting `preset_robot_class` is the simplest approach — PRC includes geometry, kinematics, and axis limits for all built-in models.

#### `Tool`

Defines a robotic end-effector tool:

| Field | Type | Description |
|---|---|---|
| `tool_type` | `FrameType` | `FIXED` (mounted on flange) or `EXTERNAL` (stationary tool in the cell) |
| `tcp` | `CartesianPosition` | Tool Center Point — the tip of the tool, relative to the robot flange. Any of the three `CartesianPosition` representations can be used. |
| `tool_id` | `string` | Unique identifier. Must match keys in `Robot.tool_dictionary` and references in `MotionGroup.tool_id`. |
| `tool_state` | `int32` | Switch between different tool configurations |
| `tool_geometry` | `PolyMesh` | Visual and collision mesh of the tool |

**Pseudocode:**
```
Tool:
    tool_type    = FIXED                // mounted on flange
    tcp          = CartesianPosition    // offset from flange to tool tip
    tool_id      = "0"
    tool_state   = 0
    tool_geometry = PolyMesh(...)       // optional visual + collision geometry
```

**Minimal tool setup (identity TCP at the flange):**

```python
tool = prc_pb2.Tool(
    tool_id="0",
    tool_type=prc_pb2.FrameType.FIXED,
    tcp=prc_pb2.CartesianPosition(
        cs=prc_pb2.CoordinateSystem(
            origin=prc_pb2.Vector3(x=0, y=0, z=0),
            x_axis=prc_pb2.Vector3(x=1, y=0, z=0),
            y_axis=prc_pb2.Vector3(x=0, y=1, z=0)
        )
    )
)
```

**Tool with Euler TCP (e.g., KUKA ZYX):**

```python
tool = prc_pb2.Tool(
    tool_id="1",
    tool_type=prc_pb2.FrameType.FIXED,
    tcp=prc_pb2.CartesianPosition(
        euler=prc_pb2.Euler(
            x=0, y=0, z=150,    # TCP offset 150mm in Z
            a=0, b=90, c=0,     # 90 degree rotation around B
            format=prc_pb2.EulerFormat.ZYX
        )
    )
)
```

#### `CustomRobot`

For defining a robot that is not in the PRC library (or modifying an existing one):

### 6.3 Motion

#### `MotionGroup`

**All motion commands must be grouped into a `MotionGroup`.** A motion group defines the interpolation type, tool, and base shared by all commands within it.

| Field | Type | Description |
|---|---|---|
| `motion_group_type` | `MotionGroupType` | `CP` (Continuous Path / Linear), `PTP` (Point-to-Point), or `SPLINE` |
| `commands` | `repeated MotionCommand` | The list of motion commands. Must be compatible with the group type. |
| `interpolation` | `string` | Interpolation mode string output in generated code (e.g., `"C_PTP"`, `"C_DIS"`). Not currently simulated. |
| `tool_id` | `string` | References a tool in `Robot.tool_dictionary` |
| `robot_base` | `Base` | Base frame for this motion group |
| `data` | `MetaData` | Additional data |

**Pseudocode:**
```
MotionGroup:
    motion_group_type = PTP | CP | SPLINE
    commands          = [MotionCommand, MotionCommand, ...]
    interpolation     = "C_PTP"           // written to generated code
    tool_id           = "0"               // must exist in Robot.tool_dictionary
    robot_base        = Base(...)
```

**Motion group type constraints:**

| MotionGroupType | Compatible Commands | Description |
|---|---|---|
| `PTP` | `AxisMotion`, `PTPMotion` | Joint-space or Cartesian point-to-point. Robot chooses fastest path. |
| `CP` | `LINMotion`, `CircularMotion` | Continuous path. TCP follows a straight line or arc in Cartesian space. |
| `SPLINE` | `PTPMotion`, `LINMotion` | Smooth spline interpolation through waypoints. |

#### `MotionCommand`

A `oneof` wrapping four motion types:

| Motion Type | Message | Description |
|---|---|---|
| `axis_motion` | `AxisMotion` | Target defined by joint angles (`JointTarget`). Use in PTP groups. |
| `ptp_motion` | `PTPMotion` | Cartesian target with PTP interpolation (`CartesianTarget`). Use in PTP groups. |
| `lin_motion` | `LINMotion` | Cartesian target with linear interpolation (`CartesianTarget`). Use in CP groups. |
| `circular_motion` | `CircularMotion` | Circular arc through 3 Cartesian positions (`repeated CartesianTarget`). Use in CP groups. |

Each motion message contains:
- `data` (`MetaData`): Optional additional data (ID, custom key-values).
- `target` / `targets`: The position target(s).

**Pseudocode — Motion Command Types:**
```
MotionCommand = ONE OF:

    AxisMotion:                           // Joint-space target
        target = JointTarget(
            axis_values = [-45, -90, 90, 0, 0, 0],  // degrees
            speed       = [0.15]                      // 15% max speed
        )

    PTPMotion:                            // Cartesian target, PTP interpolation
        target = CartesianTarget(
            position = CartesianPosition(matrix or euler or cs),
            posture  = "110",
            speed    = [0.1]
        )

    LINMotion:                            // Cartesian target, linear interpolation
        target = CartesianTarget(...)

    CircularMotion:                       // Arc through 3 Cartesian points
        targets = [CartesianTarget, CartesianTarget, CartesianTarget]
```

**Example — Axis motion (joint space):**

```python
motion = prc_pb2.MotionCommand(
    axis_motion=prc_pb2.AxisMotion(
        target=prc_pb2.JointTarget(
            axis_values=[-45, -90, 90, 0, 0, 0],  # 6 joint angles in degrees
            speed=[0.15]                            # 15% of max speed
        )
    )
)
```

**Example — Cartesian PTP motion via CoordinateSystem:**

```python
motion = prc_pb2.MotionCommand(
    ptp_motion=prc_pb2.PTPMotion(
        target=prc_pb2.CartesianTarget(
            position=prc_pb2.CartesianPosition(
                cs=prc_pb2.CoordinateSystem(
                    origin=prc_pb2.Vector3(x=500, y=0, z=800),  # mm
                    x_axis=prc_pb2.Vector3(x=1, y=0, z=0),
                    y_axis=prc_pb2.Vector3(x=0, y=1, z=0)
                )
            ),
            posture="110",
            speed=[0.1]
        )
    )
)
```

**Example — Cartesian PTP motion via Matrix4x4:**

```python
matrix = prc_pb2.Matrix4x4(
    m11=1, m12=0, m13=0, m14=0,
    m21=0, m22=1, m23=0, m24=0,
    m31=0, m32=0, m33=1, m34=0,
    m41=500, m42=0, m43=800, m44=1  # Translation: X=500mm, Y=0mm, Z=800mm
)

motion = prc_pb2.MotionCommand(
    ptp_motion=prc_pb2.PTPMotion(
        target=prc_pb2.CartesianTarget(
            position=prc_pb2.CartesianPosition(matrix=matrix),
            posture="110",
            speed=[0.1]
        )
    )
)
```

### 6.4 Actions

Actions are non-movement commands embedded in a task via `TaskPayload.action_task`:

| Action | Message | Description |
|---|---|---|
| `set_variable_action` | `SetVariable` | Sets a robot variable (bool/float/int/string) |
| `wait_for_variable_action` | `WaitForVariable` | Pauses execution until a variable reaches a target state |
| `hold_action` | `Hold` | Pauses execution for `hold_ms` milliseconds |
| `ping_action` | `Ping` | Pings the PRC server |
| `insert_code_action` | `InsertCode` | Inserts arbitrary lines of robot code. Not parsed or simulated. |

**Pseudocode:**
```
Action = ONE OF:
    SetVariable:        set variable "GripperOpen" = true
    WaitForVariable:    wait until variable "SensorReady" == true
    Hold:               pause for 2000 ms
    Ping:               ping server with payload
    InsertCode:         insert raw code lines ["LINE1", "LINE2"]
```

**Example — Hold 2 seconds then set a variable:**

```python
hold_action = prc_pb2.Action(
    hold_action=prc_pb2.Hold(hold_ms=2000)
)

set_var_action = prc_pb2.Action(
    set_variable_action=prc_pb2.SetVariable(
        new_state=prc_pb2.Variable(name="GripperOpen", boolean=True)
    )
)
```

### 6.5 Flow Control

Flow control allows branching and looping in the generated robot program.

#### `Task`

The top-level container for robot commands:

| Field | Type | Description |
|---|---|---|
| `payload` | `repeated TaskPayload` | Ordered list of command groups |
| `type` | `TaskType` | How the task is processed (see enum) |
| `name` | `string` | Human-readable task name |
| `data` | `MetaData` | Additional data |

**Pseudocode:**
```
Task:
    type    = SIMULATE_TASK | EXECUTE_TASK | EXECUTE_ON_SIMULATION_SUCCESS_TASK
              | SIMULATE_AND_EXECUTE_TASK | CONTAINER
    name    = "My Toolpath"
    payload = [TaskPayload, TaskPayload, ...]
```

**`TaskType` enum:**

| Value | Name | Description |
|---|---|---|
| 0 | `SIMULATE_TASK` | Only simulate — no execution on the real robot |
| 1 | `EXECUTE_TASK` | Execute directly without simulation |
| 2 | `EXECUTE_ON_SIMULATION_SUCCESS_TASK` | Execute only if simulation shows no errors |
| 3 | `SIMULATE_AND_EXECUTE_TASK` | Simulate then execute regardless of simulation result |
| 4 | `CONTAINER` | Groups sub-tasks without its own execution type |

#### `TaskPayload`

A `oneof` that wraps one of three command types:

| Type | Field | Description |
|---|---|---|
| `flow_task` | `Flow` | Control flow (IfElse, While, End) |
| `motion_group_task` | `MotionGroup` | A group of motion commands |
| `action_task` | `Action` | A single non-motion command |

**Pseudocode:**
```
TaskPayload = ONE OF:
    MotionGroup(...)   // most common: a group of motion commands
    Action(...)        // a single I/O or hold command
    Flow(...)          // branching or looping
```

#### `Flow`

Supports three flow types:

- **`IfElse`**: Evaluates a `Variable` condition, executes `if_true` or `if_false` `Task`.
- **`While`**: Evaluates a `Variable` condition, repeatedly executes `body` `Task` while true.
- **`End`**: Terminates the program.

**Pseudocode:**
```
Flow = ONE OF:
    IfElse:
        condition = Variable("SensorReady", boolean=true)
        if_true   = Task(... motion commands ...)
        if_false  = Task(... hold action ...)
    While:
        condition = Variable("Running", boolean=true)
        body      = Task(... repeated motion ...)
    End:
        // terminates program
```

### 6.6 Feedback

#### `SimulationResult`

Returned in `AddRobotTaskReply.simulation_result_data`. Contains the complete simulation output:

| Field | Type | Description |
|---|---|---|
| `simulation_results` | `repeated SimulationResultUnit` | One entry per interpolated position along the toolpath |
| `is_valid` | `bool` | `true` if no reachability or collision problems were detected |
| `time` | `float` | Total estimated execution time in **seconds** |
| `code` | `string` | Generated robot control code (KRL for KUKA, URScript for UR, RAPID for ABB, etc.). Write this to a file with the appropriate extension. |
| `data` | `MetaData` | Additional information |

#### `SimulationResultUnit`

One row of simulation data at a specific interpolated position:

| Field | Type | Description |
|---|---|---|
| `axis_values` | `repeated float` | Joint angles in **degrees** at this position |
| `position` | `Matrix4x4` | Cartesian TCP position at this point (mm, right-handed Z-up) |
| `time` | `float` | Time in **seconds** from simulation start to this point |
| `collision` | `repeated bool` | One per axis — `true` if that axis is in collision. Only active with collision checking enabled. |
| `singularity` | `repeated bool` | One per axis — `true` if the robot is near a singularity |
| `outofreach` | `repeated bool` | One per axis — `true` if that axis is out of its valid range |
| `external_axis_values` | `repeated float` | External axis positions at this point |
| `external_axis_outofreach` | `repeated bool` | Out-of-range flags for external axes |
| `interpolation_factor` | `float` | Indicates if this is an interpolated point (0.0) or an original command point (1.0) |
| `id` | `string` | Matches the `MetaData.id` of the original motion command |
| `alarm` | `bool` | Aggregated flag — `true` if **any** collision or out-of-reach condition exists |

**Pseudocode:**
```
SimulationResultUnit:
    axis_values          = [A1, A2, A3, A4, A5, A6]  // degrees
    position             = Matrix4x4                   // TCP in mm, world frame
    time                 = 1.25                        // seconds from start
    collision            = [false, false, ...]          // per axis
    singularity          = [false, false, ...]          // per axis
    outofreach           = [false, true, ...]           // per axis
    external_axis_values = [E1, E2, ...]
    interpolation_factor = 1.0   // 1.0 = original point, 0.0 = interpolated
    id                   = "cmd_0"
    alarm                = true  // any issue detected
```

#### `RobotState`

Used for real-time visualization. Returned by `GetSimulatedRobotState` and streamed via `SubscribeRobotFeedback`:

| Field | Type | Description |
|---|---|---|
| `axis_position` | `JointTarget` | Current joint angles in degrees |
| `robot_transformations` | `repeated TransformationArray` | **Key for visualization.** First array = robot joints (base through flange). Subsequent arrays = external axes. Each `TransformationArray` contains one `Matrix4x4` per joint. |
| `toolpath_index` | `float` | Normalized position along the toolpath (0.0–1.0) |
| `tool_id` | `string` | Currently active tool ID |

## 9. SimulationResult (AddTaskReply Data)

The `SimulationResult` returned from `AddRobotTask` provides complete simulation feedback:

### Top-Level Fields

| Field | Value | Description |
|---|---|---|
| `is_valid` | `bool` | `true` if the entire toolpath is reachable and collision-free |
| `time` | `float` | Total estimated execution time in seconds |
| `code` | `string` | Generated robot code (KRL, URScript, RAPID, etc.) |
| `simulation_results` | `list` | Per-position simulation data |

### Per-Position Data (`SimulationResultUnit`)

Each unit represents one interpolated step along the toolpath:

**Pseudocode:**
```
SimulationResultUnit:
    axis_values:          [A1, A2, A3, A4, A5, A6]  // degrees
    position:             Matrix4x4                   // TCP in mm, world frame
    time:                 float                       // seconds from start
    collision:            [bool, bool, bool, ...]     // per axis
    singularity:          [bool, bool, bool, ...]     // per axis
    outofreach:           [bool, bool, bool, ...]     // per axis
    external_axis_values: [E1, E2, ...]               // mm or degrees
    external_axis_outofreach: [bool, ...]
    interpolation_factor: float    // 1.0 = original point, 0.0 = interpolated
    id:                   string   // matches input command MetaData.id
    alarm:                bool     // true if ANY issue detected
```

### Processing Example

```python
result = reply.simulation_result_data

if not result.is_valid:
    print("WARNING: Toolpath has issues!")

for i, unit in enumerate(result.simulation_results):
    if unit.alarm:
        issues = []
        if any(unit.outofreach):
            issues.append("OUT_OF_REACH")
        if any(unit.collision):
            issues.append("COLLISION")
        if any(unit.singularity):
            issues.append("SINGULARITY")
        print(f"Position {i} (t={unit.time:.2f}s): {', '.join(issues)}")
        print(f"  Axes: {list(unit.axis_values)}")

# Save generated code to file
with open("robot_program.src", "w") as f:  # .src for KRL, .script for UR
    f.write(result.code)
```

## 10. RobotState & Visualization

The `RobotState` message is the primary mechanism for visualizing the robot. It can be obtained in two ways:
1. **Polling:** Call `GetSimulatedRobotState` with a `normalized_state` (0.0–1.0).
2. **Streaming:** Receive via `SubscribeRobotFeedback` (when `stream_update=True` or during real-time control).

### TransformationArray — What It Represents

The `robot_transformations` field is a `repeated TransformationArray`. Each `TransformationArray` contains a list of `Matrix4x4` transformations.

`RobotState.robot_transformations[0]` contains **one `Matrix4x4` per robot link** (base + each joint), in order:

| Index | Link |
|---|---|
| 0 | Base |
| 1 | Axis 1 |
| 2 | Axis 2 |
| 3 | Axis 3 |
| 4 | Axis 4 |
| 5 | Axis 5 |
| 6 | Flange |

Subsequent entries in `robot_transformations` contain external axis transforms (if any):

**Pseudocode:**
```
robot_transformations[0]  →  Robot body
  .transformation[0]  →  Base transform              (Matrix4x4, world-space, mm)
  .transformation[1]  →  Axis 1 transform
  .transformation[2]  →  Axis 2 transform
  .transformation[3]  →  Axis 3 transform
  .transformation[4]  →  Axis 4 transform
  .transformation[5]  →  Axis 5 transform
  .transformation[6]  →  Flange transform            (tool geometry attached here)

robot_transformations[1]  →  External axis 1 (if present)
  .transformation[0..N]  →  External axis joint transforms

robot_transformations[2]  →  External axis 2 (if present)
  ...
```

**For a standard 6-axis robot, `robot_transformations[0]` contains 7 matrices** (base + 6 axes). Each matrix is a **world-space** transformation in the PRC coordinate system (right-handed, Z+ up, millimetres). It describes **where and how that link's mesh is positioned and oriented in world space** at the current robot state.

### Applying Transforms — Mesh Positioning vs. Joint Positions

The transformation matrices are designed to **transform each link mesh from its rest pose to its current world pose**. This is the primary use case for visualization.

**Important:** The matrices do **not** directly give joint pivot positions. To find the world position of a joint's pivot point, you must apply the matrix to that joint's rest-frame origin using the row-vector convention:

```
world_pos = [x, y, z, 1] @ M

wx = x * m11 + y * m21 + z * m31 + m41
wy = x * m12 + y * m22 + z * m32 + m42
wz = x * m13 + y * m23 + z * m33 + m43
```

Where `(x, y, z)` is the joint's pivot point in the robot's rest/home frame.

#### Rest-Frame Origins — Example: KUKA KR AGILUS KR610R11002

These are the joint pivot points in the robot's home/rest frame (mm). They are **robot-model-specific** and are derived from the robot's geometry definition — they are not returned by the PRC server at runtime:

```python
AGILUS_ORIGINS = [
    (  0,  0,    0),   # Base
    ( 25,  0,  400),   # A1
    ( 25,  0,  960),   # A2  (400 + 560)
    ( 25,  0,  985),   # A3  (400 + 560 + 25)
    (540,  0,  985),   # A4  (25 + 515)
    (630,  0,  985),   # A5  (25 + 515 + 90)
]
```

To compute the world position of, say, the A2 pivot:
```
origin_A2 = (25, 0, 960)
M_A2      = robot_transformations[0].transformation[2]

world_x = 25 * M_A2.m11 + 0 * M_A2.m21 + 960 * M_A2.m31 + M_A2.m41
world_y = 25 * M_A2.m12 + 0 * M_A2.m22 + 960 * M_A2.m32 + M_A2.m42
world_z = 25 * M_A2.m13 + 0 * M_A2.m23 + 960 * M_A2.m33 + M_A2.m43
```

### Flange & TCP Position — Direct Access

Unlike link transforms, the `RobotState.flange_frame` and `RobotState.tool_frame` fields are **already world-space frames**. Their origin **is** the flange/TCP position. Read it directly — no local-offset multiplication is needed:

```
flange_world_pos = (state.flange_frame.m41,
                    state.flange_frame.m42,
                    state.flange_frame.m43)

tcp_world_pos    = (state.tool_frame.m41,
                    state.tool_frame.m42,
                    state.tool_frame.m43)
```

### Visualization Algorithm

**Pseudocode — Core Visualization Loop:**
```
// Setup phase: load robot joint meshes (one PolyMesh per joint)
joint_meshes = load_robot_geometry()    // 7 PolyMeshes for 6-axis robot
tool_meshes  = load_tool_geometry(active_tool_id)

// Per-frame update:
function update_visualization(robot_state):
    transforms = robot_state.robot_transformations[0].transformation

    // Apply transform to each joint mesh
    for i = 0 to num_joints:
        matrix = transforms[i]               // Matrix4x4, world-space, mm
        matrix = convert_to_platform(matrix) // apply unit/handedness conversion
        joint_meshes[i].set_world_transform(matrix)

        // Color override for alarms
        if robot_state.axis_alarm[i] == true:
            joint_meshes[i].set_color(RED)
        else:
            joint_meshes[i].restore_original_color()

    // Apply flange transform to tool
    flange_matrix = transforms[num_joints]   // last transform = flange
    flange_matrix = convert_to_platform(flange_matrix)
    for mesh in tool_meshes:
        mesh.set_world_transform(flange_matrix)

    // External axes (if any)
    for ext_idx = 1 to len(robot_state.robot_transformations):
        ext_transforms = robot_state.robot_transformations[ext_idx].transformation
        for j, matrix in ext_transforms:
            external_axis_meshes[ext_idx][j].set_world_transform(
                convert_to_platform(matrix)
            )
```

### Platform-Specific Coordinate Transforms

**All PRC matrices are in a right-handed, Z+ up coordinate system with millimetre units.** Different platforms require different conversions:

| Platform | Extra Steps |
|---|---|
| **Python / NumPy** | Direct — mm, Z+ up, row-major. No conversion needed. |
| **Rhino / Grasshopper** | Direct — mm, Z+ up, row-major. No conversion needed. |
| **Blender** | Divide translation by 1000 (mm → m), then `matrix.transpose()` (Blender is column-major). |
| **Fusion 360** | Divide translation by 10 (mm → cm), use `setWithArray` with column-major order. |
| **Unity** | Divide by 1000, swap Y and Z, flip handedness (left-handed Y-up). |
| **Babylon.js** | Divide by 1000, rotate meshes 90 deg around X at load, set `scene.useRightHandedSystem = true`. |

#### Rhino / Grasshopper (Right-handed, Z-up, mm)
**No conversion needed.** Direct mapping from PRC `Matrix4x4` to Rhino `Transform`:

```csharp
Transform xform = new Transform();
xform.M00 = matrix.M11; xform.M01 = matrix.M12; xform.M02 = matrix.M13; xform.M03 = matrix.M14;
xform.M10 = matrix.M21; xform.M11 = matrix.M22; xform.M12 = matrix.M23; xform.M13 = matrix.M24;
xform.M20 = matrix.M31; xform.M21 = matrix.M32; xform.M22 = matrix.M33; xform.M23 = matrix.M34;
xform.M30 = matrix.M41; xform.M31 = matrix.M42; xform.M32 = matrix.M43; xform.M33 = matrix.M44;
```

#### Blender (Right-handed, Z-up, **metres**)
Blender uses metres, so **divide translation by 1000**. Blender also expects column-major matrices, so **transpose**:

```python
for j, xform in enumerate(robot_state.robot_transformations[0].transformation):
    matrix = mathutils.Matrix([
        [xform.m11, xform.m12, xform.m13, xform.m14],
        [xform.m21, xform.m22, xform.m23, xform.m24],
        [xform.m31, xform.m32, xform.m33, xform.m34],
        [xform.m41/1000, xform.m42/1000, xform.m43/1000, xform.m44]
    ])
    matrix.transpose()
    bpy.data.objects[f"Axis_{j}"].matrix_world = matrix
```

#### Fusion 360 (Right-handed, Z-up, **centimetres**)
Fusion 360 uses centimetres, so **divide translation by 10**. The `Matrix3D.setWithArray` expects a **column-major** layout:

```python
for j, xform in enumerate(robot_state.robot_transformations[0].transformation):
    transform = adsk.core.Matrix3D.create()
    transform.setWithArray([
        xform.m11, xform.m21, xform.m31, xform.m41 / 10,  # Column 0
        xform.m12, xform.m22, xform.m32, xform.m42 / 10,  # Column 1
        xform.m13, xform.m23, xform.m33, xform.m43 / 10,  # Column 2
        xform.m14, xform.m24, xform.m34, xform.m44         # Column 3
    ])
```

#### Unity (Left-handed, Y-up, **metres**)
Unity requires both a **coordinate system swap** (right-handed Z-up to left-handed Y-up) and **unit conversion** (mm to m). Multiply coordinates by 1000 when sending to PRC, divide by 1000 when receiving:

```
PRC (right-handed Z-up, mm)  →  Unity (left-handed Y-up, m)
  PRC_X  →  Unity_X
  PRC_Y  →  Unity_Z
  PRC_Z  →  Unity_Y
  PRC_mm →  Unity_m  (/ 1000)
```

The `PRC_Unity.cs` file handles this conversion. See the Unity integration for the full transformation code.

#### JavaScript / Babylon.js (Right-handed, Y-up, **metres**)
Babylon.js scene must be set to right-handed (`scene.useRightHandedSystem = true`). Translation values divided by 1000. Initial mesh rotation of 90 deg around X to convert from Z-up to Y-up:

```javascript
// Initial mesh setup (once)
for (let mesh of scene.meshes) {
    mesh.rotation.x = Math.PI / 2;
    mesh.bakeCurrentTransformIntoVertices();
}

// Per-frame update
for (let mesh of scene.meshes) {
    var axisIndex = parseInt(mesh.name.charAt(2));
    var xf = xForms[axisIndex];
    var matrix = BABYLON.Matrix.Zero();
    matrix.setRowFromFloats(0, xf.getM11(), xf.getM12(), xf.getM13(), xf.getM14());
    matrix.setRowFromFloats(1, xf.getM21(), xf.getM22(), xf.getM23(), xf.getM24());
    matrix.setRowFromFloats(2, xf.getM31(), xf.getM32(), xf.getM33(), xf.getM34());
    matrix.setRowFromFloats(3, xf.getM41()/1000, xf.getM42()/1000, xf.getM43()/1000, xf.getM44());
    mesh.getWorldMatrix().copyFrom(matrix);
}
```

### Sending Cartesian Positions FROM Host Application to PRC

When converting positions from a host application (e.g., Blender object transforms) to PRC `CartesianPosition`, the inverse conversion is needed. For Blender (metres to mm), **multiply translation by 1000**:

```python
matrix = obj.matrix_world.copy()
matrix.transpose()  # Blender column-major → PRC row-major
prc_matrix = prc_pb2.Matrix4x4(
    m11=matrix[0][0], m12=matrix[0][1], m13=matrix[0][2], m14=matrix[0][3],
    m21=matrix[1][0], m22=matrix[1][1], m23=matrix[1][2], m24=matrix[1][3],
    m31=matrix[2][0], m32=matrix[2][1], m33=matrix[2][2], m34=matrix[2][3],
    m41=matrix[3][0]*1000, m42=matrix[3][1]*1000, m43=matrix[3][2]*1000, m44=matrix[3][3]
)
```

### Alternative Visualization: Axis Values

If your platform supports forward kinematics or you have robot geometry rigged with joint rotations, you can use `axis_position.axis_values` directly instead of the transformation matrices. The values are in **degrees** for each joint.

## 11. Mesh & Geometry System

PRC uses a hierarchical mesh system for robot visualization and collision checking.

### Hierarchy

**Pseudocode:**
```
Robot
├── geometry (built-in for preset robots)
│   ├── PolyMesh[0]  →  Base (joint 0)
│   │   ├── meshes[0..N]                  → N visualization meshes (different colors)
│   │   ├── collision_convex_hull[0..M]   → M collision meshes (simplified)
│   │   ├── transform                     → initial transform (Matrix4x4)
│   │   └── name                          → "Base"
│   ├── PolyMesh[1]  →  Axis 1
│   │   └── ...
│   ├── PolyMesh[2]  →  Axis 2
│   │   └── ...
│   └── ... (one PolyMesh per axis)
├── tool_dictionary
│   └── Tool.tool_geometry  →  PolyMesh for the tool
│       ├── meshes[0..N]
│       └── collision_convex_hull[0..M]
├── collision_geometry  →  Environment obstacles
│   └── repeated PolyMesh
└── cell_geometry  →  Work cell structure
    └── PolyMesh
```

### Mesh Color Convention

Each `Mesh` has a single `mesh_color` field of type `Int4`:

```
Int4 { x=A, y=R, z=G, w=B }
```

Where A, R, G, B are integers 0–255. Examples:
- Opaque red: `{ x=255, y=255, z=0, w=0 }`
- Semi-transparent gray: `{ x=128, y=155, z=155, w=155 }`
- Default (no color set): `{ x=255, y=155, z=155, w=155 }`

### PolyMesh Structure

A `PolyMesh` is a compound object that may contain:
- **Multiple visualization meshes** (`meshes`): Each with its own faces, vertices, normals, and color. A single robot joint might be rendered as several colored parts.
- **Multiple collision meshes** (`collision_convex_hull`): Simplified convex hulls for collision checking. These are separate from the visualization meshes and may differ in number and detail.
- **A transform** (`Matrix4x4`): Initial placement relative to WorldXY.
- **A name** (`string`): Identifier.

**Important:** The count of `meshes` and `collision_convex_hull` within a single `PolyMesh` are **independent**. A PolyMesh might have 5 visual meshes but 1 collision hull, or 2 visual meshes and 3 collision hulls.

### Building Meshes for Sending to PRC

When defining tool geometry or collision objects:

```python
mesh = prc_pb2.Mesh(
    vertices=[
        prc_pb2.Vector3(x=0, y=0, z=0),
        prc_pb2.Vector3(x=100, y=0, z=0),
        prc_pb2.Vector3(x=100, y=100, z=0),
        prc_pb2.Vector3(x=0, y=100, z=0),
    ],
    faces=[
        prc_pb2.Int4(x=0, y=1, z=2, w=3),  # Quad face
        prc_pb2.Int4(x=0, y=1, z=2, w=2),  # Triangle face (z==w)
    ],
    mesh_color=prc_pb2.Int4(x=255, y=100, z=100, w=100)  # ARGB gray
)

polymesh = prc_pb2.PolyMesh(
    meshes=[mesh],
    collision_convex_hull=[mesh],  # Same mesh for collision (or simplified version)
    name="MyObject"
)
```

## 12. Coordinate Systems & Unit Conventions

### PRC Internal Convention

| Property | Value |
|---|---|
| Coordinate system | **Right-handed** |
| Up axis | **Z+** |
| Linear units | **Millimetres (mm)** |
| Angular units | **Degrees** |
| Matrix layout | Row-major (`System.Numerics.Matrix4x4` compatible) |
| Translation in matrix | Bottom row: m41 (X), m42 (Y), m43 (Z) |

### Platform Conversion Summary

| Platform | Handedness | Up | Units | Translation Scale | Matrix Notes |
|---|---|---|---|---|---|
| **Rhino/GH** | Right | Z+ | mm | 1:1 | Direct mapping |
| **Blender** | Right | Z+ | m | / 1000 | Transpose (column-major) |
| **Fusion 360** | Right | Z+ | cm | / 10 | Column-major `setWithArray` |
| **Unity** | Left | Y+ | m | / 1000 + swap Y<->Z | Axis swap + handedness flip |
| **Babylon.js** | Right* | Y+ | m | / 1000 | Rotate meshes 90 deg around X at load |

\* Babylon.js requires `scene.useRightHandedSystem = true`.

### Euler Format by Robot Brand

| Robot Brand | Euler Format | Convention |
|---|---|---|
| KUKA | `ZYX` (value 0) | Rotations: Z, then Y, then X |
| Universal Robots | `AXISANGLE` (value 1) | Rotation vector = axis * angle |
| NEURA | `RPY` (value 2) | Roll-Pitch-Yaw |
